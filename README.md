因为：
  1. LLM 有上下文限制
  2. LLM 上下文越长生成质量越差，幻觉越多
  3. LLM 并不能始终全力以赴，而是会越来越偷懒

又因为：
  1. 特定提示词与知识库下的 LLM 适合完成特定领域的工作
  2. LLM 会逐渐偏移原始目标，适合完成直接，描述详细的工作
  3. LLM 是黑盒，必须在实践中评估其水平，并加以监督

还因为：
  1. 不同 LLM 模型对不同方面本身擅长程度不同
  2. 前沿高级 LLM 模型的价格非常昂贵，指数增长

所以：
  请你实现这个名为 “LaYER” 的 Agent 系统，由上层 Agent 不断产出出新的更下层 Agent，
  并采取市场化社会化竞争原则，每个任务都需要有至少多个下层 Agent 竞争，
  竞争者需要提供他们完成该任务的大致计划，并给出关键部分的详细描述，
  竞争者不会并行制定他们的方案，而是会参考之前所有 Agent 的计划来给出有批判性的新计划；
  上层 Agent 将审批这些竞争者的方案，并选出多个个方案不同的 Agent 同时进行工作，
  并选择结果最好的作为最终成果。所有 Agent 被设为非常懒惰，如果竞标者给出的方案几乎一致，
  他们才会自己去执行而不是交给下层执行。系统有一个多层 Agent 池，Agent 配置采用会插值到 Agnet 配置中，比如 `copilot -s --model {{model}} -p {{prompt}}`。
  请你保持项目结构的透明性，简洁性，用最直接的方法实现。

运行流程：
  Agent 0 将初始任务拆分成多个子任务,产生 Layer 1，每个子任务为单独并行运行的竞争池
    -> 每个竞争池创建至少 N 个 Agent 1 提交逐个方案，当新 Agent 0 认为 Agent 1 提出的方案不再与之前所有方案具有显著差异时停止
      -> 初始 Agent 0 选出至少 M 个方案，在各自的 Git Worktree 中并行实施各自的方案
        -> 初始 Agent 0 检查所有人的结果，选出最好的结果，合并 Worktree
  Agent 1 收到 Agent 0 给出的某个子任务，继续拆分
    -> 递归 ... 
  Agent X 收到 Agent X-1 给出的某个子任务，继续拆分
    -> 达到配置限制，自己实现
    -> 没达到配置限制，但是所有 Agent X+1 给出的方案几乎完全一致，自己实现

全局配置最多 Agent 数量限制，防止资源耗尽。
